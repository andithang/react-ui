# Design Pattern

This page defines the design system contract for this library: **tokens**, **semantic color roles**, and **type scales** used by components.

## 1) Core principles

- **Token-first**: components consume CSS variables, not raw values.
- **Semantic naming**: tokens describe purpose (`--ui-control-border-hover`) rather than one-off appearance.
- **Theme aware**: light (`:root`) and dark (`[data-theme='dark']`) provide matching semantic keys.
- **Type-safe usage**: TS type unions (sizes, statuses, variants) model allowed API values.

### 3.2 Dark control behavior notes

- Control borders remain neutral/high-contrast (`--ui-control-border*`), not primary-tinted.
- Option hovers/selects still use primary-tinted semantic background tokens.
- Primary button remains brand `#e03` with ant-style hover/active intensities.

## 4) Type system and scales

### 4.1 Shared spacing type

`src/styles/tokens.ts` exports:

- `spaceTokenMap`: `{ none, xs, sm, md, lg, xl, 2xl }`
- `SpaceScale` type: `keyof typeof spaceTokenMap`

### 4.3 Color role usage guidance

- **Text/labels**: `--ui-text`, `--ui-text-muted`
- **Surfaces**: `--ui-bg`, `--ui-surface`, `--ui-surface-raised`
- **Borders/focus**: `--ui-control-border*`, `--ui-focus*`
- **Interactive states**: `--ui-primary*`, `--ui-control-option-hover-bg`, `--ui-control-option-selected-bg`
- **Semantic feedback**: `--ui-danger`, `--ui-success`, `--ui-warning`

## 5) Contract checklist for new components

1. Use semantic tokens (no hardcoded one-off colors where token exists).
2. Provide light + dark correctness through existing semantic keys.
3. Prefer `className` extension and `ui-` prefixed class namespace.
4. Keep states (`hover`, `active`, `focus-visible`, `disabled`, `error`) token-driven.
5. Add Storybook examples for variant/size/state coverage.
